# Project Agent Guide

## Product

Scan Release List is a bilingual (French/English) React application for
aggregating manga and webtoon releases. Users configure source pages and CSS
selectors, the TanStack Start server fetches those pages, and the browser parses
and displays the results in a compact, responsive list.

The application currently supports:

- source creation, editing, duplication, enable/disable, and deletion;
- live source previews and up to three chapter selector rows;
- configurable date formats and optional server-side cover proxying;
- newest-first results, virtualized/infinite rendering, and visited history;
- reversible hidden releases with a three-second undo window;
- selective JSON import/export of sources, hidden items, and visit history;
- persistent light/dark themes and French/English localization.

## Stack and Commands

- React 19, TypeScript, TanStack Start/Router, and Vite
- Tailwind CSS 4 and shadcn/ui (Base UI primitives)
- Dexie/IndexedDB for browser persistence
- i18next for localization
- Vitest, Testing Library, ESLint, and Prettier
- pnpm (`pnpm-lock.yaml` is authoritative)

Use these commands:

```bash
pnpm install
pnpm dev          # http://localhost:3000
pnpm test
pnpm typecheck
pnpm lint
pnpm check        # verify formatting
pnpm build
pnpm preview
```

Before delivery, run at least `pnpm test`, `pnpm typecheck`, `pnpm lint`, and
`pnpm build`. Run `pnpm check` when source formatting may have changed. For UI
changes, also exercise the affected flow in a browser at desktop and mobile
widths when feasible.

`src/routeTree.gen.ts` is generated by TanStack Router. Do not edit it manually.

## Architecture Boundaries

- Components render state and dispatch user actions. Keep persistence, parsing,
  and transfer logic out of components.
- Hooks coordinate reactive UI data. Dexie live queries belong in hooks such as
  `src/hooks/use-release-data.ts`.
- Services/repositories own scraping, parsing, persistence, and import/export.
- The Dexie database definition and migrations live under `src/lib/db`.
- Shared domain types live under `src/types`, with filenames ending in
  `.type.ts`. Do not declare reusable domain types in components.
- Keep parsing and normalization functions small and pure. Test edge cases
  without requiring network access.
- Preserve the `@/` alias for imports from `src`.

The scan flow is:

1. `ReleaseDashboard` reads configured sources through Dexie live queries.
2. Enabled sources are passed to `scanReleaseSource`.
3. `fetchSourceHtml` fetches remote HTML in a TanStack Start server function.
4. `parseReleaseHtml` parses it in the browser using configured selectors.
5. The dashboard merges and sorts results, while hidden and visited state comes
   from IndexedDB.

## Data and Identity

Dexie database name: `scan-release-list`.

- `sources`: source configuration records.
- `hiddenReleases`: reversible hidden manga identities.
- `visitedReleases`: chapter visits and recent-history metadata.

Dexie is the source of truth for persisted user data; scan results themselves
are transient React state. Import uses validated, versioned JSON and upserts
records rather than clearing the database.

Preserve stable identities:

- manga item: source ID plus normalized manga URL, falling back to title;
- chapter visit: source ID plus release URL;
- parsed chapter: release URL, label, and optional time label.

When changing persisted records, add a Dexie schema version/migration where
needed, update runtime import guards, export types, repositories, and tests.
Deleting a source must also clean up all source-owned persisted records. Verify
this invariant when adding new tables.

## Scraping and Security

- Treat all remote HTML and imported JSON as untrusted.
- Missing optional selectors or malformed elements must skip only the affected
  item. Invalid configured CSS selectors should produce a clear source error,
  not crash the application.
- Normalize relative links with `new URL(value, baseUrl)` and accept only
  `http:`/`https:` inputs.
- Keep response limits in place (currently 4 MB HTML and 2 MB images).
- Never render scraped HTML with `dangerouslySetInnerHTML`.
- Never persist secrets or add secrets to source configurations.
- Server fetch functions currently permit arbitrary HTTP(S) URLs. Before a
  public/untrusted deployment, add SSRF protections (block loopback, private,
  link-local, and cloud metadata destinations; validate redirects and resolved
  addresses), timeouts, and appropriate rate limiting.
- Optional image proxying must validate that the response is actually an
  allowed image type before returning it.

This is client-side-first, but scanning and image proxying require a TanStack
Start server runtime. Do not describe the current build as a static-only app.

## TypeScript and Tests

- Keep strict TypeScript clean; avoid `any`, unchecked casts, and broad
  `Record<string, unknown>` compatibility workarounds in new code.
- Use explicit persisted and parsed domain types.
- Prefer discriminated unions for multi-state UI flows.
- Add focused unit tests next to logic as `*.test.ts` or `*.test.tsx`.
- Parsing tests should cover missing nodes, relative/invalid URLs, invalid CSS,
  duplicate structures, and supported date formats.
- Persistence tests should cover IDs, retention/expiry, cascade cleanup, and
  import validation.
- UI tests should assert accessible behavior and persisted preferences rather
  than implementation details.

## UI and Localization

- Use existing shadcn/ui primitives before introducing custom controls.
- Use semantic Tailwind tokens (`background`, `foreground`, `muted`,
  `destructive`, etc.) rather than raw palette colors where a token or variant
  exists.
- Compact actions use icon buttons with visible tooltips and accessible labels.
- Keep the release list dense, scannable, keyboard-accessible, and responsive.
- Preserve virtualization for large result sets.
- Hidden state must remain reversible and hidden releases stay excluded by
  default.
- All user-visible text belongs in both `src/lib/i18n/locales/fr.ts` and
  `src/lib/i18n/locales/en.ts`. Use `translateError` for known service errors.
- Theme and language preferences are stored in localStorage; avoid hydration
  flashes or mismatches when changing their initialization.

## Git Workflow

- `main` is the stable base; `develop` is the integration branch.
- Branch from `develop` using `feature/<description>`, `fix/<description>`, or
  `refactor/<description>`.
- Merge topic branches into `develop` with `--no-ff`.
- Make small, focused commits and do not mix unrelated cleanup into a change.
- Preserve user work in a dirty worktree; never discard unrelated edits.

If the requested work starts on another branch, inspect the current state before
switching or creating branches and avoid rewriting shared history.
